<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Generated public/data to dated snapshot | Static JSON and entity detail files are copied into rollback/audit snapshots. Incomplete copies could create false rollback confidence. |
| `latest.json` aggregate data to entity detail files | Frontend drill-down routes trust that every aggregate entity has a matching static detail JSON file. |
| Data Pipeline workflow_dispatch to Deploy Pages checkout | Dispatch metadata crosses workflow boundaries and must bind deployment to the generated-data commit, not a moving branch tip. |
| Deploy Pages workflow to GitHub Pages | Validated static files are built and published with Pages/OIDC permissions. |
| GitHub Actions runner environment | Workflow tokens and inputs are available during CI and must not expand privileges or leak secrets. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-06-01 | Tampering | `scripts/snapshot-data.ts` snapshot contents | mitigate | Recursively copy `public/data/entities` into `snapshots/{snapshotId}/entities` inside the existing copy try/catch and delete partial snapshot directories on copy failure. |
| T-04-06-02 | Repudiation | `public/data/snapshots/manifest.json` | mitigate | Record `entities/` in `SnapshotManifestEntry.files` when detail files are copied so snapshot completeness is auditable. |
| T-04-06-03 | Tampering | `scripts/validate-data.ts` latest-to-detail link | mitigate | Require `entities/` when `latest.json.entities` is non-empty and check every valid latest entity path `entities/{type}/{id}.json` exists before deployment. |
| T-04-06-04 | Denial of Service | Static frontend detail routes | mitigate | Fail `npm run data:validate` before build/deploy when missing detail files would cause static 404s for drill-down/detail pages. |
| T-04-06-05 | Spoofing | Deploy Pages `workflow_dispatch` `source_sha` | mitigate | Checkout `inputs.source_sha` when provided and verify checked-out HEAD equals the requested SHA before validation/build/publish. |
| T-04-06-06 | Tampering | Deploy Pages branch race | mitigate | Use `ref: ${{ inputs.source_sha || github.sha }}` so data-pipeline dispatches build the generated commit instead of whichever `main` commit is current at checkout time. |
| T-04-06-07 | Elevation of Privilege | `.github/workflows/deploy.yml` permissions | mitigate | Preserve `contents: read`, `pages: write`, `id-token: write`; add no `contents: write`, `actions: write`, PATs, repository_dispatch, or secrets. Contract tests assert no secrets and least privilege. |
| T-04-06-08 | Information Disclosure | Workflow logs | mitigate | Verification step prints only expected/actual commit SHAs, not tokens or secrets; Deploy Pages workflow continues to reject `secrets.` usage. |
| T-04-06-09 | Denial of Service | Snapshot retention pruning | accept | Existing retention logic prunes only manifest-listed old snapshots and preserves root data; this plan does not widen deletion scope. Existing tests continue to cover retention safety. |
</threat_model>
