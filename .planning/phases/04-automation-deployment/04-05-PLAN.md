Required handoff shape for this gap closure:
```yaml
# data-pipeline.yml target state
permissions:
  contents: write
  actions: write

- name: Commit generated data
  id: commit-generated-data
  run: |
    git add public/data
    if git diff --cached --quiet; then
      echo "No generated data changes to commit"
      echo "pushed=false" >> "$GITHUB_OUTPUT"
    else
      git commit -m "data: refresh automated snapshot"
      git push
      echo "pushed=true" >> "$GITHUB_OUTPUT"
    fi

- name: Dispatch Deploy Pages
  if: steps.commit-generated-data.outputs.pushed == 'true'
  env:
    GH_TOKEN: ${{ github.token }}
  run: |
    gh workflow run deploy.yml --ref main \
      -f source=data-pipeline \
      -f source_run_id="${{ github.run_id }}" \
      -f source_sha="$(git rev-parse HEAD)"
```

```yaml
# deploy.yml target state
on:
  push:
    branches: [main]
  workflow_dispatch:
    inputs:
      source:
        description: 'Dispatch source, e.g. data-pipeline'
        required: false
        type: string
      source_run_id:
        description: 'Data Pipeline run id that requested deployment'
        required: false
        type: string
      source_sha:
        description: 'Generated data commit SHA requested for deployment'
        required: false
        type: string
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Data Pipeline workflow to repository main branch | Contents-write workflow commits and pushes generated static JSON to `main`. |
| Data Pipeline workflow to GitHub Actions workflow_dispatch API | Data workflow uses `GITHUB_TOKEN`/`github.token` with `actions: write` to trigger Deploy Pages. |
| Deploy Pages workflow to GitHub Pages | Pages/OIDC workflow validates/builds committed static data and publishes `dist`. |
| External scraper/staged data to public static site | Failed or no-fresh scraper output must not promote, snapshot, commit, or deploy stale/bad data. |
| Generated public/data push to workflow triggers | Generated commits must not recursively trigger Data Pipeline. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05-01 | Spoofing | workflow_dispatch handoff | mitigate | Use only `${{ github.token }}` inside the originating Data Pipeline workflow; do not introduce PATs or secrets. Dispatch only `deploy.yml` on `--ref main` with source metadata inputs. |
| T-04-05-02 | Tampering | data-pipeline.yml repository permissions | mitigate | Add only `actions: write` to existing `contents: write`; do not grant Pages/OIDC or broad extra permissions to the data workflow. Contract tests assert permissions. |
| T-04-05-03 | Repudiation | deployment handoff audit trail | mitigate | Pass `source=data-pipeline`, `source_run_id=${{ github.run_id }}`, and `source_sha=$(git rev-parse HEAD)` to Deploy Pages workflow_dispatch so the run records its source. |
| T-04-05-04 | Denial of Service | recursive workflow triggering | mitigate | Keep `public/data/**` absent from Data Pipeline push paths and assert no command dispatches `data-pipeline.yml`; handoff targets only Deploy Pages. |
| T-04-05-05 | Tampering | failed/no-fresh data preservation | mitigate | Dispatch step is conditional on `steps.commit-generated-data.outputs.pushed == 'true'` and occurs only after staged validation, promotion, snapshot, root validation, build, bundle check, commit, and push have succeeded. |
| T-04-05-06 | Information Disclosure | workflow tokens/secrets | mitigate | Use no `secrets.*` values and do not print tokens; `GH_TOKEN` is scoped to the runner environment for `gh workflow run` only. |
| T-04-05-07 | Elevation of Privilege | deploy workflow | accept | Deploy workflow already has least-privilege `contents: read`, `pages: write`, `id-token: write`; this plan only adds optional dispatch inputs and does not add write access or secrets. |
</threat_model>
